Abstract

MEDISLOT is a comprehensive, full-stack web application developed on the MERN (MongoDB, Express.js, React.js, Node.js) technology stack, designed to digitise the complete lifecycle of medical appointment booking, doctor-patient communication, online payment processing, and emergency blood donation coordination within a single, unified healthcare ecosystem.


The system serves three distinct user roles — Patients, Doctors, and Administrators — through three independently deployable sub-applications: a patient-facing frontend, a shared admin/doctor panel, and a RESTful backend API server with a co-located Socket.IO WebSocket server.

Chapter 4 — Payment Integration (Razorpay)
4.1 Payment Architecture Overview
Razorpay is integrated as the primary payment gateway. The implementation follows the industry-standard Order → Checkout → Server-side Verification pattern, ensuring payment cannot be spoofed by a malicious client.

4.5 Security Considerations
✅ Server-Side Verification: Payment is never trusted based on the client's claim. The backend independently fetches the order from Razorpay's API (orders.fetch()) and checks status === 'paid' before updating the database or sending the receipt email.
⚠️ Amount Validation: The amount sent from the frontend is validated against the appointment's actual fees. The Razorpay order amount is set in paise (amount * 100), which Razorpay enforces independently.

Chapter 5 — Real-Time Chat System (Socket.IO)
5.1 Chat Architecture Overview
The chat module is built on Socket.IO v4.8, co-located with the Express server via http.createServer(app). This allows WebSocket and HTTP to share the same port. Messages are persisted to MongoDB for cross-session retrieval while real-time delivery is handled via WebSocket events.

5.4 Chat Message Persistence Flow
1
User opens /doctor/chat → ChatList.jsx calls POST /api/chat/rooms with all possible roomIds. The backend aggregates: messageModel.aggregate([$match, $sort, $group]) to return only rooms with ≥1 message + last message preview.
2
User clicks a doctor → ChatWindow.jsx fetches history: GET /api/chat/messages/userId_docId → messageModel.find({roomId}).sort({timestamp:1}). All historical messages rendered in chronological order.
3
Socket joins room: socket.emit("join_room", roomId) and announces presence: socket.emit("user_online", {roomId, name, role})
4
User types and sends: socket.emit("send_message", {...}). Server persists to MongoDB then broadcasts: socket.to(roomId).emit("receive_message", ...)
5
When doctor opens the chat: socket.emit("mark_seen", {roomId, readerName}). Server bulk-updates all unseen messages and emits messages_seen → patient sees read receipts.

Chapter 11 — Conclusion & Future Enhancements
11.1 Conclusion

The Medislot project has successfully delivered a production-grade, full-stack healthcare management web application addressing the complete patient-doctor-administrator workflow within a unified digital platform. The following objectives were achieved:

✅ Achievements:
  • Multi-step, security-hardened authentication with bcrypt, OTP verification, JWT cookies, and cryptographic reset tokens
  • Real-time slot-based appointment booking with atomic slot locking preventing double-booking
  • Full Razorpay payment integration with server-side verification and automated receipt emails
  • Production-ready Socket.IO chat with MongoDB persistence, presence tracking, and read receipts
  • Geospatial blood donor matching engine using MongoDB $geoNear + ABO/Rh compatibility matrix + multi-channel notification (socket + email)
  • 10+ automated transactional email types via Brevo API
  • Complete Admin and Doctor panels with analytics, CRUD, and emergency broadcast capability
  • Production deployment on Vercel with CORS, cookie security, and environment configuration
